# `conn.go` 改动说明（面向阅读者，使用伪代码）

## 概述

这份说明面向不直接修改代码的读者，解释最近对 `src/server/conn.go` 做出的改善目的、思路和验证方法。为便于理解，核心实现使用伪代码描述，去掉了语言特有的细节。

## 要解决的问题（简要）

- TCP 传输会把一条逻辑消息拆成多个数据块（分片），直接逐块处理会导致半条消息被误解析。
- 有些客户端/终端使用非 UTF‑8 编码（常见于 Windows 的 GB18030），直接按 UTF‑8 解析会出现乱码或解析失败。
- 非常长的单条消息会耗尽内存或导致处理阻塞。
- 连接异常或处理 panic 时需要确保资源（协程、计数器）能被正确回收。

## 设计思路（要点）

- 按“逻辑行”重组数据：当读取到一个行片段时，先暂存，直到确认该行的最后一个片段到达再做整体处理。
- 编码兼容：优先按 UTF‑8 解析，不通过则尝试常见的 GB18030 编码，再兜底成文本展示。
- 长消息保护：设定一个合理的长度上限，超过则丢弃并向客户端返回提示，防止资源耗尽。
- 活动心跳与超时回收：在处理循环中向外层通知“我还活着”，若长时间未收到活动信号则关闭连接。
- 容错性：处理入口用捕获机制（recover）防止 panic 导致整个连接管理崩溃，并保证清理信号被发送。

## 核心伪代码（易读版）

读取并处理每一条逻辑消息：

```
常量 MAX_LEN = 65536  // 举例

循环直到连接关闭:
    设置读超时（例如 5 分钟）
    parts = []
    total = 0

    当行未结束:
        chunk, isPrefix, err = 读取一段行数据()
        如果 err 是 EOF 或网络错误: 关闭连接并退出

        total += len(chunk)
        如果 total > MAX_LEN:
            // 丢弃本条剩余数据直到行结束
            如果 isPrefix:
                一直读取并舍弃直到 isPrefix == false
            回复客户端："消息长度超限"
            跳到外层循环（继续下一条消息）

        parts.append(chunk)
        如果 isPrefix == false:
            raw = 合并(parts)
            text = 解码文本(raw)  // 见下
            如果 text 非空:
                如果 text 看起来像 JSON 协议:
                    解析并交给协议处理器
                否则:
                    当作普通文本消息处理
                向上层发送 活动信号
            结束当前消息处理

```

文本解码（伪码）：

```
函数 解码文本(bytes):
    如果 bytes 是合法 UTF-8: 返回 UTF-8 字符串
    否则 尝试按 GB18030 解码并验证为 UTF-8, 成功则返回
    否则 返回最接近的文本表示（尽量不崩溃）
```

活动心跳与超时（外层管理）：

```
启动 一个管理线程:
    增加连接计数
    启动 上面的读取循环（在子线程）
    isLive = 带缓冲的信号通道

    循环:
        如果 从 isLive 收到关闭信号 或 通道被关闭: 退出并清理
        如果 超过 N 秒未收到活动信号:
            发送通知给客户端: "你被踢了"
            关闭连接并退出
```

建议在发送活动信号时使用非阻塞方式，以避免当外层管理阻塞时内部处理线程被挂起。

## 为什么对使用 `nc` 的场景有效

- `nc` 常用于手动发送行数据，且在不同系统与网络条件下，单条逻辑消息可能会被拆开发送。以上“片段重组”保证只有在完整行就绪后再处理，从根本上避免半包解析问题。
- 对非 UTF‑8 的输入回退到 GB18030 可以显著减少在 Windows 平台手动测试时遇到的乱码或解析失败。
- 长消息保护防止误用或攻击导致服务不可用。

## 验证建议（简单，可由运维或 QA 执行）

- 手动用 `nc` 发送短文本和简单 JSON，确认服务有响应。示例：
  - 使用 `nc localhost 8888` 输入 `hello`，观察服务器回复或日志。
- 模拟分片：写一个小脚本分多次发送一条很长的行，确认服务器能正确拼接或在超长时提示丢弃。
- 编码验证：用脚本发送 GB18030 编码的中文，看服务器是否仍能识别文本内容。

## 后续建议（可选）

- 在活动信号发送处使用非阻塞写入（以避免潜在阻塞）。
- 将 `MAX_LEN`、读超时、踢出超时时间等作为可配置项，便于生产环境调整。
- 增加运行时指标（例如被丢弃的消息数、解码失败数、活动连接数），便于监控与告警。

## 小结

本次改动着眼于提升对手工/测试工具（如 `nc`）输入场景的健壮性，解决了消息半包、编码不兼容、超长消息与异常回收等常见问题。文档以伪代码呈现，便于非代码维护人员理解实现思路与验证方法。
